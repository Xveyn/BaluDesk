🥧 BaluPi

Low-Power Raspberry Pi Cache-Tier für das BaluHost-Ökosystem — immer erreichbar, energieeffizient, optimiert für den 80/20-Zugriff.

Raspberry Pi 3B+ 24/7 • ~4W Hot-Tier Cache FastAPI (Minimal)
Motivation

Warum BaluPi?

Das NAS (BaluHost) verbraucht ~65W im Idle. Für die alltäglichen 20% der Dateien — Fotos, Dokumente, aktive Projekte — braucht es keinen Ryzen-Server. Ein Raspberry Pi mit SSD reicht als Always-On-Cache-Tier.

Architektur

Gesamtsystem-Übersicht

BaluPi integriert sich als Cache-Tier zwischen Clients und NAS. Clients fragen zuerst den Pi, der Pi leitet bei Bedarf an das NAS weiter.

Kernprinzip

80/20 — Smart Tiering statt Full Mirror

Der Pi spiegelt nicht das gesamte NAS. Stattdessen hält er nur die Dateien, die aktiv genutzt werden. Das spart Speicher, Bandbreite und hält die Komplexität niedrig.

📸

Mobile ➜ Pi First

BaluApp (Android) synct Fotos und Dokumente immer zuerst auf den Pi. Der Pi ist per VPN erreichbar, verbraucht kaum Strom und queued alles für's NAS.

  • Kamera-Backup landet sofort auf Pi-SSD
  • Dokumente (PDF, Office) werden gecacht
  • Pi synct asynchron zum NAS (wenn online)
  • Kein NAS-Wake nötig für Alltagsnutzung
🖥️

Desktop ➜ Auto-Detect

BaluDesk erkennt automatisch, ob NAS oder Pi verfügbar ist (mDNS). Für kleine Dateien bevorzugt er den Pi, für große direkt das NAS.

  • Dateien < 50MB → Pi (schnell, immer da)
  • Dateien > 50MB → NAS direkt (USB 2.0 Bottleneck)
  • Pi-Cache wird als "Prefetch" genutzt
  • Offline-Queue wenn beides nicht erreichbar
💡 Angepasste Strategie: Statt den Pi als vollständigen NAS-Spiegel zu betreiben (was die 1 GB RAM und USB 2.0 überfordert), wird er als intelligenter Hot-Cache eingesetzt. Das reduziert die Implementierungskomplexität um ~40% — kein Full-Metadata-Sync nötig, nur die relevanten Dateien + ihre Metadaten.
Sync-Strategie

Mobile-First: BaluApp → BaluPi → BaluHost

Der optimierte Sync-Flow für die häufigste Nutzung: Fotos und Dokumente vom Smartphone.

1
BaluApp nimmt Foto / erstellt Dokument

Kamera-Backup oder manueller Upload. Datei wird in lokale Upload-Queue eingereiht.

2
Upload an BaluPi (immer verfügbar)

Per VPN/LAN → Pi REST-API. Datei landet auf SSD. Metadaten in SQLite. Status: pending_nas_sync

3
Pi synct zum NAS (wenn online)

Periodischer Check oder Handshake-Trigger. Alle pending_nas_sync Dateien werden hochgeladen. Status wird synced.

4
Cache-Eviction auf Pi

Sobald zum NAS gesynct und Cache-Limit erreicht: älteste bereits-gesynchte Dateien werden vom Pi entfernt. Metadaten bleiben.

Reverse-Flow (NAS → Pi → Client): Wenn BaluDesk eine Datei anfragt, die nur auf dem NAS liegt und der Pi sie nicht hat, gibt der Pi im Active Mode ein Redirect/Proxy an das NAS. Im Passive Mode meldet er file_not_cached — der Client zeigt dann den Metadaten-Eintrag aber ohne Download-Option.
Handshake-Protokoll

Mode Transition: Active ↔ Passive

Der kritischste Teil des Systems: Was passiert, wenn das NAS aufwacht oder sich abschaltet?

🟢 NAS wacht auf (Passive → Active)

  • Pi erkennt NAS via mDNS/Heartbeat
  • Pi sendet: last_sync_ts + pending_changes[]
  • NAS empfängt Uploads, merged Metadaten
  • NAS sendet: Änderungen seit last_sync_ts
  • Pi aktualisiert lokalen Cache
  • Clients werden über Mode-Switch notifiziert
  • Pi wird Proxy, NAS wird Primary

🔴 NAS fährt runter (Active → Passive)

  • NAS sendet Shutdown-Broadcast
  • Pi empfängt aktuellen Metadaten-Snapshot
  • Pi bestätigt Readiness
  • NAS fährt runter
  • Pi wird Primary für alle Clients
  • Neue Uploads → Pending-Queue
  • Nicht-gecachte Dateien: unavailable
⚠️ Edge Case — Unerwarteter NAS-Ausfall: Wenn das NAS ohne Shutdown-Broadcast ausfällt, erkennt der Pi dies nach 3 fehlgeschlagenen Heartbeats (~30s) und wechselt automatisch in Passive Mode. Beim nächsten NAS-Start wird ein Full Reconciliation Handshake durchgeführt (dauert länger, prüft alle Metadaten).
API-Design

Gleiche Oberfläche, angepasster Scope

BaluPi exponiert die gleichen REST-Endpunkte wie BaluHost, sodass Clients nicht wissen müssen, mit wem sie sprechen. Nicht verfügbare Features geben 501 Not Available in Passive Mode zurück.

Endpunkt BaluHost BaluPi Anmerkung
/api/auth/* ✅ Full ✅ Read-Only Spiegel User-DB wird bei Handshake synchronisiert
/api/files/upload ✅ Full ✅ In Queue Uploads landen auf SSD, werden gequeued für NAS
/api/files/download ✅ Full ⚠️ Nur Cache Active: Proxy zum NAS · Passive: nur gecachte Files
/api/files/list ✅ Full ✅ Metadaten Zeigt alle Dateien (mit cached/not-cached Flag)
/api/sync/* ✅ Full ✅ Lokal Sync gegenüber Pi-Cache statt NAS-Storage
/api/system/* ✅ RAID, SMART ❌ Pi hat kein RAID. Pi-eigenes /api/pi/health
/api/backups/* ✅ Full ❌ Backups nur auf dem NAS
/api/shares/* ✅ Full ⚠️ Cached only Sharing nur für Dateien im Cache
/api/mobile/* ✅ Full ✅ Primary Mobile Registrierung + Sync direkt auf Pi
/api/pi/* — 🆕 Neu status, cache, wake-nas, sync-queue
Datenhaltung

Storage-Layout auf dem Pi

/mnt/ssd/     # SATA SSD (ext4, Journaling)
├── cache/            # Hot-Tier Dateien (LRU Eviction)
│   ├── user_1/       # Pro-User Verzeichnis
│   │   ├── photos/   # Kamera-Backup
│   │   └── docs/     # Dokumente
│   └── user_2/
├── pending/          # Noch nicht zum NAS gesynct
├── db/               # SQLite (WAL-Modus)
│   ├── balupi.db    # Metadaten, Cache-Index
│   └── users.db     # User-Spiegel (read-only)
└── config/           # BaluPi Konfiguration

SQLite statt PostgreSQL

Pi 3B+ hat nur 1 GB RAM. PostgreSQL würde allein ~100-200 MB fressen. SQLite im WAL-Modus bietet gleichzeitige Reads + sequentielle Writes — perfekt für den Use-Case.

Cache-Eviction Policy

Hybrid-LRU: Dateien, die vom User als "Always Available" markiert sind, werden nie evicted. Alles andere nach Last-Access-Time, wenn das Cache-Limit (konfigurierbar, z.B. 80% SSD) erreicht ist. Nur bereits zum NAS gesynchte Dateien werden evicted.

Client-Anpassungen

Was ändert sich für BaluApp & BaluDesk?

📱

BaluApp (Android) — Anpassungen

  • Sync-Target Priorität: Pi > NAS > Offline-Queue
  • Auto-Discovery: mDNS findet Pi (immer) und NAS (wenn an)
  • Kamera-Backup: Geht immer zuerst an Pi
  • Upload-Größe-Check: >100 MB → direkt NAS (wenn da)
  • Status-Anzeige: "Synced to Pi" / "Synced to NAS"
  • VPN: Bestehende WireGuard-Integration reicht
🖥️

BaluDesk (Desktop) — Anpassungen

  • Server-Discovery: Bestehender mDNS-Code erweitern
  • Dual-Target Sync: HTTP-Client lernt Failover Pi↔NAS
  • UI: Neuer Status-Indikator (Pi/NAS/Offline)
  • Große Dateien: Direkt an NAS (Pi-USB 2.0 Limit)
  • Settings: "Prefer Pi" Toggle im Settings-Panel
  • Remote-Server-Profile: Pi als Profil hinzufügbar
🔑 Entscheidende Vereinfachung: Da die API-Oberfläche gleich bleibt, sind die Client-Anpassungen minimal. Im Kern ist es nur ein neuer Service-Discovery-Layer + Routing-Logik (welches Target bekommt welchen Upload). Die Sync-Engine, Conflict-Resolution und Auth bleiben unverändert.

Hardware-Realität

Raspberry Pi 3B+ — Grenzen kennen

Ressource Pi 3B+ Implikation Mitigation
CPU 4× A53 @ 1.4 GHz FastAPI mit 1-2 Uvicorn Workers Async I/O, kein Heavy Processing
RAM 1 GB SQLite statt PostgreSQL Streaming Uploads, kein In-Memory-Cache
USB USB 2.0 (max ~35 MB/s) SSD-Bottleneck Große Dateien direkt ans NAS routen
Netzwerk ~300 Mbit/s effektiv Reicht für Fotos/Docs Ausreichend für 80/20-Nutzung
Strom ~4W 24/7 betreibbar für ~8€/Jahr —
💡 Upgrade-Empfehlung: Falls die Performance im Praxistest nicht reicht, ist ein Pi 4 (2 GB+) ein Drop-In Replacement: USB 3.0 (~400 MB/s SSD), echtes Gigabit-Ethernet, 2-8 GB RAM. Der Pi 3B+ ist ein valider Prototyp — aber für Dauerbetrieb mit mehreren gleichzeitigen Clients wäre ein Pi 4/5 besser.
Implementierung

Phasenplan (angepasst an 80/20)

Durch den Fokus auf Hot-Cache statt Full-Mirror reduziert sich die Komplexität deutlich. Der Mobile-First-Ansatz wird priorisiert.

P0

Pi Setup & Proof of Concept

Raspberry Pi OS Lite, SSD-Mount (ext4), Python 3.11, FastAPI Skeleton. Benchmark: SSD-Throughput, RAM-Nutzung, Netzwerk.

Aufwand: 3-4 Tage Prio: Kritisch
P1

BaluPi Minimal-Backend

Fork von BaluHost-Backend. SQLite-Adapter. Nur Core-Endpunkte: Auth (read-only Spiegel), File Upload/Download, Sync, Mobile-API. Kein RAID, kein SMART, kein Scheduler.

Aufwand: 2-3 Wochen Prio: Hoch
P2

Mobile-First: BaluApp → Pi Sync

BaluApp lernt Pi-First-Routing. Kamera-Backup → Pi. Service Discovery (mDNS). Upload-Queue mit Priority (Pi > NAS). Status-Anzeige im App.

Aufwand: 2 Wochen Prio: Hoch Deps: P1
P3

Handshake-Protokoll & NAS-Sync

Pi ↔ NAS Bidirectional Sync. Mode Detection (Active/Passive). Pending-Queue Upload. Metadaten-Merge. Conflict Resolution. Heartbeat-Monitoring.

Aufwand: 3-4 Wochen Prio: Mittel Deps: P1 ⚠️ Komplexeste Phase
P4

Cache-Layer & Eviction

LRU Eviction Policy. "Always Available" Markierung. Cache-Größen-Management. BaluDesk Smart-Routing (große Dateien → NAS direkt).

Aufwand: 1-2 Wochen Prio: Mittel Deps: P3
P5

Frontend-Anpassung & Hardening

WebApp: Active/Passive Mode UI. BaluDesk: Server-Discovery Update. Testing aller Flows (Mobile→Pi→NAS, Pi-Ausfall, NAS-Ausfall). Stress-Tests auf Pi 3B+.

Aufwand: 2-3 Wochen Prio: Mittel Deps: P3, P4
Gesamt: ~10-14 Wochen
~30% weniger als Full-Mirror Ansatz
Mobile-Flow nach P2 testbar
Risiken & Mitigations

Worauf achten?

⚠️

Split-Brain Szenario

NAS und Pi haben unterschiedliche Versionen derselben Datei (z.B. NAS separat gestartet + direkt bearbeitet).

  • Lösung: Vector Clocks oder Lamport Timestamps pro Datei
  • Beim Handshake: Conflict Detection auf File-Ebene
  • Worst Case: Keep-Both (Rename mit .conflict)
🔒

Security: Physischer Zugriff

Pi ist physisch leichter zugänglich als ein NAS im Schrank. Jemand könnte die SSD abziehen.

  • Lösung: LUKS-Verschlüsselung auf SSD
  • JWT-Secrets verschlüsselt speichern
  • User-Passwörter nie auf dem Pi (nur Token)
💾

SSD-Wear auf dem Pi

Ständige Writes (Uploads, SQLite WAL) belasten die SSD. Consumer SSDs haben begrenzte TBW.

  • Lösung: SSD mit hoher TBW wählen (>200 TBW)
  • Write Amplification minimieren (batch writes)
  • SMART-Monitoring auf dem Pi einrichten
🔌

Stromausfall / Korruption

Pi hat kein USV. Plötzlicher Stromausfall kann Daten korrumpieren.

  • Lösung: ext4 Journaling + SQLite WAL
  • Write-Barrier aktiviert
  • Optional: USB-USV (~15€) für 5 Min. Puffer
Fazit

Empfehlung

Der 80/20-Ansatz mit Mobile-First macht BaluPi deutlich machbarer als ein Full-Mirror. Der Pi wird nicht zum Mini-NAS, sondern zum Smart-Cache — genau das, wofür der Pi 3B+ mit seinen Limitierungen gut geeignet ist.

✅ Machbar mit Pi 3B+
📱 Mobile-First Sync ab Phase 2
⚡ ~120€/Jahr Stromersparnis
Nächster Schritt: Phase 0 starten — Pi aufsetzen, SSD mounten, FastAPI Skeleton deployen. Dann einen realistischen Benchmark fahren: Wie schnell ist Upload/Download über USB 2.0? Wie viel RAM bleibt bei laufendem Uvicorn? Das liefert die Daten für die Go/No-Go Entscheidung zum Pi 3B+ vs. Upgrade auf Pi 4.
